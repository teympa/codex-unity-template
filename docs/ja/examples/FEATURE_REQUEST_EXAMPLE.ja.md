# 依頼文の例

これは、初心者が実装前に Codex へ渡しやすい feature request の例です。

## 例

```text
機能: Enemy stun system

目的:
重い攻撃を受けた敵が一定時間スタン状態になるようにしたいです。

期待する挙動:
- 重い攻撃でスタンが発生する
- スタン中は移動と攻撃ができない
- スタン中は見た目の表示を出してよい
- スタン終了後は通常状態へ戻る

制約:
- gameplay ロジックは Assets/Scripts/Gameplay に置く
- UI や演出は gameplay のルール判定と分ける
- スタン時間や閾値はコードに直書きしない
- 状態遷移は明示する

依頼:
- まず design-review をしてほしい
- 構造が良さそうならその後に実装してほしい
```

## この依頼文が良い理由

- 目的がはっきりしている
- 挙動と実装詳細が分かれている
- フォルダ責務が書かれている
- state 遷移を明示する前提がある
- 調整値の直書きを避ける意図がある

## その後に Codex がやると良い流れ

1. 目的、変更対象、リスクを短く要約する
2. design-review を行う
3. state、調整値、UI 更新の置き場を提案する
4. 小さく実装する
5. 実装後に軽い検査を回す
